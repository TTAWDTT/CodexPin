- [ ] 步骤 1：初始化 Electron 项目骨架  
  要求：使用 npm 初始化项目并安装 Electron 作为开发依赖；在仓库中创建 `electron/main.js`、`electron/preload.js`、`renderer/index.html`、`renderer/style.css`、`renderer/renderer.js` 等基础文件；`package.json` 中配置好 `"main"` 字段和 `"start"` 脚本指向 Electron 入口。  
  验收指标：在项目根目录执行 `npm run start`，能够成功启动 Electron，打开一个加载 `renderer/index.html` 的空白基础窗口且无报错。

- [ ] 步骤 2：实现始终置顶的无边框小部件窗口  
  要求：在 `electron/main.js` 中通过 `BrowserWindow` 配置创建一个尺寸适合的小窗口（例如宽约 340–380，高约 200–260），启用 `alwaysOnTop: true`、`frame: false`，并配置 `webPreferences.preload` 指向 `preload.js`；在渲染层添加可拖拽区域以支持拖动窗口。  
  验收指标：启动应用后，小部件窗口始终悬浮在其他应用之上，无系统标题栏和边框，可通过顶部区域拖动移动位置，关闭窗口不会产生异常错误日志。

- [ ] 步骤 3：搭建 CodexPin 面板 UI 结构与基础样式  
  要求：在 `renderer/index.html` 中搭建顶栏（展示会话时间与周剩余额度）和主体状态区（3–4 行状态文本）的 DOM 结构；在 `renderer/style.css` 中使用 Flex 布局和基础颜色、间距完成初步排版，但暂不要求最终视觉精细；面板整体呈现为单一矩形区域。  
  验收指标：启动应用后，可以看到包含顶栏和多行文本区的基础界面，占满小窗口，结构清晰，元素不重叠、不溢出，可通过开发者工具验证 DOM 结构符合设计。

- [ ] 步骤 4：实现会话与周度预算的前端状态管理逻辑  
  要求：在渲染层实现独立的状态模块（如 `renderer/state.js`），定义 `Session` 和 `WeeklyBudget` 数据结构以及 `statusLines` 队列；提供 `startSession`、`stopSession`、`setIdle`、`setPhase`、`appendStatusLine`、`getState` 等接口，并在 `renderer/renderer.js` 中根据状态实时更新 DOM；实现按秒更新的计时器，只在会话激活时累加时间。  
  验收指标：在代码中调用状态接口启动和停止会话时，顶栏时间显示能正常从 0 递增并在停止后保持不变；模拟多次会话后，周已用时间和剩余时间显示符合手工计算结果；追加多条状态文本时界面只保留最近 3–4 行。

- [ ] 步骤 5：打通 JSON 持久化存储（会话与周度预算）  
  要求：在主进程和 preload 中通过 IPC 提供 `loadState` 与 `saveState` 能力，在用户目录的应用数据路径下读写单一 JSON 状态文件；在渲染层启动时从持久化数据中恢复周度预算和必要的会话摘要，并在关键状态变更（停止会话、更新周度数据等）时触发保存。  
  验收指标：首次运行应用执行若干会话后退出再重启，周已用时间和剩余时间与上次退出前保持一致；删除状态文件后重启会自动回到干净初始状态；读取或写入失败时不会导致应用崩溃，而是记录错误并安全退回默认状态。

- [ ] 步骤 6：实现窗口位置和显示模式（完整/极简）的持久化  
  要求：在主进程中监听窗口移动与关闭事件，记录窗口的最新位置和尺寸，并通过状态文件持久化；在渲染层实现完整模式（显示顶栏和状态区）与极简模式（仅显示顶栏）的切换逻辑，例如通过双击顶栏切换，同时将当前模式写入持久化状态。  
  验收指标：用户拖动小部件到屏幕任意位置后关闭应用并重新启动，窗口会在同一位置重新出现；切换到极简模式后再次重启仍保持极简模式；模式切换和位置保存不会引发窗口尺寸异常或闪烁。

- [ ] 步骤 7：完成 霞鹜文楷字体集成与黑色磨砂玻璃视觉风格  
  要求：在项目中引入霞鹜文楷字体文件并通过 `@font-face` 在 `renderer/style.css` 中注册，将小部件主字体设置为霞鹜文楷并提供合理的后备字体；调优背景为黑色磨砂玻璃风格，包含半透明深色背景、模糊和柔和阴影，使用字号、字重和明暗区分顶栏与状态内容的层级。  
  验收指标：在目标机器上启动应用，小部件整体采用霞鹜文楷显示且无明显锯齿或渲染异常；背景呈现半透明模糊的玻璃质感，顶栏文字明显比状态文本更突出，但整体风格统一、易读、不刺眼。

- [ ] 步骤 8：完善低额度提示与待命状态显示逻辑  
  要求：在渲染逻辑中根据周剩余时间动态应用低额度样式，例如当剩余时间少于 60 分钟时改变顶栏剩余时间的颜色或添加轻微强调效果；在会话处于待命状态时，左侧时间明确显示为 0 或「待命」，且不会继续累加会话时间。  
  验收指标：通过模拟多次长会话将周剩余时间压缩至 60 分钟以下时，界面能够明显提示预算偏低；切换到待命状态后时间显示稳定为 0 或指定文案，重新切回激活状态后计时从上次停止时刻重新开始。

- [ ] 步骤 9：编写并执行核心功能测试与手动验收清单  
  要求：根据 `docs/solution.md` 中的设计要点整理一份最小测试清单，覆盖会话计时正确性、周度预算累积与重置、状态行队列行为、持久化恢复、窗口置顶与拖动、模式切换以及低额度提示等关键路径，并实际执行这些检查（可为脚本测试或严格的手动测试流程）。  
  验收指标：所有预期行为在测试过程中均能按照设计实现，无明显逻辑错误或界面异常；对于发现的问题已修复并重新跑通对应测试用例；测试结果有简单记录（例如在 `docs/` 下添加一份简要测试记录或在现有文档中标注完成）。

- [ ] 步骤 10：完成打包与文档更新  
  要求：为 Electron 应用配置至少一个目标平台的打包流程（如使用 `electron-builder`），在 `package.json` 中添加相应构建脚本并能生成可分发安装包或可执行文件；更新 `README.md` 或新增使用说明文档，简要说明 CodexPin 的功能、安装方式、启动方式以及已知限制。  
  验收指标：在本机成功运行打包命令生成安装包或可执行文件，并在全新环境中安装后能正常启动小部件且行为与开发模式一致；项目根目录的文档清楚说明如何运行和使用 CodexPin，使未参与开发的使用者也能独立完成安装与启动。
